# Modules Interaction, API, and Manifests

## 1) Общая модель

В системе каждый bounded context работает как отдельный модуль (отдельный service в Docker Compose) со своим API и своей БД.  
Модули не читают чужую БД напрямую: интеграция идет только через HTTP API и через настройки связей в `plugin-registry`.

## 2) Роль plugin-registry

`plugin-registry` отвечает за:

- хранение подключенных модулей;
- получение manifest от модулей (self-hosted `GET /manifest`);
- управление связями модулей (`/plugins/_links`);
- выдачу metadata для UI (`/plugins`, `/plugins/{name}`).

`ui-shell` использует эти данные, чтобы:

- показывать модуль в меню только если он подключен/активен;
- показывать межмодульные шаги (например, Contacts в Orders) только если включена связь `orders -> contacts`.

## 3) Manifest: зачем и как

Manifest — это контракт модуля с экосистемой.  
Каждый модуль отдает `GET /manifest` и описывает:

- `name`, `bounded_context`, `version`;
- UI metadata (названия меню, конфигурация экранов);
- API metadata (`api.base_url`);
- при необходимости event metadata.

Важно: manifests хранятся в самих модулях (self-hosted), а не централизованно в одном каталоге.

## 4) API-интеграция между модулями

Базовые принципы:

- только HTTP API между bounded contexts;
- идентификация связей сущностей — через UUID;
- отсутствие fallback к прямым SQL-запросам в чужие БД;
- если связь модуля не включена — UI/flow явно сообщает, что шаг недоступен.

Пример: `orders` для шага контакта обращается в `contacts` API (по manifest-конфигурации endpoint), а не в таблицы `contacts` напрямую.

## 4.1) Как UI ходит в API

`ui-shell` в браузере использует same-origin путь `/api`, который проксируется nginx (`ui-shell-https`) в `gateway`.

Почему это важно:

- не нужно обращаться к отдельному порту gateway из браузера;
- меньше проблем с CORS/TLS в HTTPS окружении;
- все запросы модулей (`/plugins`, `/orders`, `/contacts`, `/finance` и т.д.) идут через единый вход.

## 5) Текущий flow (Orders -> Contacts)

1. В `orders/create` пользователь проходит шаги выбора заказа.  
2. На шаге "Далее" UI проверяет связь `orders -> contacts` через `plugin-registry`.  
3. UI получает manifest `contacts` и берет из него endpoints/fields для блока контактов.  
4. По номеру телефона выполняется API-поиск контакта в `contacts` service.  
5. При выборе найденного контакта данные подставляются в форму заказа.

Это делает интеграцию модульной: можно менять/подключать модули без жесткой связки на уровне БД.
